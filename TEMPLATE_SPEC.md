# Community Templates — Structure & Metadata Spec

## Repository Structure

Templates use a flat structure sorted alphabetically. Each template lives in its own directory under `templates/` with version subdirectories.

```
templates/
  aws-cloudwatch-docker-wrapper/
    meta.yaml          ← fully auto-generated, do not edit manually
    dashboard.png      ← optional screenshot
    5.0/
      README.md
      template.xml
    6.0/
      README.md
      template.yaml
  aws-vpn-http/
    meta.yaml
    7.0/
      README.md
      template.yaml
  elasticsearch/
    meta.yaml
    8.0/
      README.md
      template.yaml
```

Contributors only need to provide: template folder, version subdirectory, `README.md`, template YAML/XML, and optionally `dashboard.png`.

## `meta.yaml`

Fully auto-generated by the GitHub Action — **do not edit manually**. All fields are derived or fetched automatically.

```yaml
# derived from README / YAML
title: "AWS CloudWatch via Docker Wrapper"
description: "..."          # first paragraph of README (latest version)
author: "..."               # ## Author section of README (latest version)
maintainer: "..."           # seeded from author on first run; updated via claim-maintainer / clear-maintainer labels
zabbix_tested_up_to: "6.0" # max version folder name
tested_with: "..."          # parsed from README best-effort
category: []                # see Categories section — open question
screenshot: true            # true if dashboard.png exists in template folder
versions: ["5.0", "6.0"]   # all version subdirectory names
type: template              # template | webhook | dashboard | other — derived from top-level key in YAML **NEEDS CHECKING! Alternatively split integrations into folders by type at highest directory level 
type: template              # template | webhook | dashboard | other — derived from top-level key in YAML

# computed daily by GitHub Action
status: active              # active | update_needed | broken
maintained: true            # true if maintainer field is set
weight: 280
last_updated: "2026-05-01"
open_issues: 2
discussion_count: 8
```

> All README and YAML fields are parsed from the **latest version subdirectory** (highest version number).

## Supported Zabbix Versions

Maintained by the Zabbix team in `_data/supported_versions.json`. Updated when a new Zabbix version is released or end-of-life.

```json
["6.4", "7.0", "7.2", "8.0"]
```

A template is considered version-compatible if at least one of its version subdirectories matches an entry in this list.

## Status

Computed daily by the GitHub Action. Two independent dimensions:

### `status`

Single value, priority order: `broken` > `update_needed` > `active`

| Status | Condition |
|--------|-----------|
| `active` | At least one version subdirectory matches `supported_versions.json` |
| `update_needed` | `update_needed` label applied by Action |
| `broken` | `broken` label applied by Action |

### `maintained`

//maybe not needed, just use filter option by maintainer name or unmaintained

Derived from `meta.yaml` — `true` if `maintainer` field is set, `false` if empty.

## Labels

### User-created

| Label | Purpose |
|-------|---------|
| `broken` | Report that a template doesn't work |
| `outdated` | Template works but needs update for newer Zabbix version |
| `claim-maintainer` | Request to take ownership of a template |
| `clear-maintainer` | Maintainer stepping down |
| `new-template` | PR adding a new template |
| `enhancement` | Improvement to existing template |

### Zabbix team only

| Label | Purpose |
|-------|---------|
| `needs-review` | Team triage label |

---

## Automation Rules

### Action auto-applies labels

| Trigger | Action |
|---------|--------|
| `broken` or `outdated` issue open + no maintainer response in **14 days** | Apply `update_needed` label to template |
| No version subdirectory matches `supported_versions.json` | Apply `update_needed` label to template |
| `broken` issue open + no maintainer response in **60 days** | Apply `broken` label to template |

### Requires Zabbix team (never automated)

//I doubt we need approved-maintainer, seems overengineering, can set manually.

| Trigger | Team action |
|---------|-------------|
| `broken` label applied | Review and optionally clear `maintainer` in `meta.yaml` |
| `clear-maintainer` issue opened | Action notifies team; team clears `maintainer` in `meta.yaml` |
| `claim-maintainer` issue opened | Team vets request and manually sets `maintainer` in `meta.yaml`, closes issue |

> Clearing the maintainer field always requires Zabbix team approval — it is never done automatically.

## Weight / Ranking

Cards on the portal are sorted by weight — higher weight appears first.

### Official templates

Weight is manually set in `meta.yaml`:

```yaml
weight: 1000
```

Official templates always appear before community templates (community score caps below 400).

### Community templates

Weight is **auto-computed daily** by the GitHub Action. Never set manually in `meta.yaml`.

```
weight = (discussion_count × 5, capped at 150)
       + (min(closed_issues / (open_issues + 1), 4) × 20, capped at 80)
       + (has_maintainer ? 100 : 0)
       + (has_screenshot ? 50 : 0)
       - (status == "update_needed" ? 100 : 0)
       - (status == "broken" ? 999 : 0)
```

| Signal | Max contribution | Rationale |
|--------|-----------------|-----------|
| Discussions (capped) | +150 | Popularity proxy |
| Issue stability ratio (capped) | +80 | Problems get fixed = reliable |
| Has maintainer | +100 | Someone is responsible for it |
| Has screenshot | +50 | Proxy for effort and quality |
| update_needed penalty | −100 | May not work on current Zabbix |
| Broken penalty | −999 | Hard demotion, never surfaces near top |

**Max score: 380. With `update_needed`: max 280. Official templates start at 1000+.**

## GitHub Action

Runs daily. For each template:

1. Parses README and template YAML from latest version subdirectory
2. Fetches live stats from GitHub API (issues, discussions, last commit)
3. Checks versions against `_data/supported_versions.json`
4. Checks for `dashboard.png` screenshot
5. Computes status and weight
6. Applies labels where rules are met
7. Notifies Zabbix team where human action is required
8. Writes updated `meta.yaml` per template (single commit: "chore: daily stats update")

Authentication uses the auto-provided `GITHUB_TOKEN` — no setup required.

> **Note:** GitHub disables scheduled workflows after 60 days of repository inactivity.
